# Agent Flow 预览日志节点抽象

规则：Agent Flow 预览 / 对话日志里的追踪节点，需要复用完整的“可展开节点项”抽象，而不是只复用节点行内容；route / fusion 内部 LLM trace 子树里的根节点和分支 LLM 节点也属于同类追踪节点，只是在树形层级上新增缩进和子节点，不应另做静态卡片或独立折叠样式；fusion 只是执行分组，不是合成的 LLM 父节点，不能渲染成 fake LLM / LLM2 上级；分支 LLM 详情必须消费后端提供的 node-run-like `input_payload` / `debug_payload` / `output_payload`，复用原有输入、数据处理、输出三段，不让前端从 summary 或 raw JSON 推断；展开详情槽位只放输入、数据处理、输出等运行详情，不要再重复渲染节点名 / 节点类型头部；节点行里的节点名和耗时 / token 指标应保持在同一行；耗时超过 1000ms 后自动进位显示为秒；查看详情只表达展开态，不要额外套选中态颜色；对话日志打开窗口必须使用画布共享 side dock 外壳，保留宽度拖拉能力。

原因：用户明确指出“抽象不完整，不支持展开这些”，后续又指出展开后“重复了”、指标“不要放到下面，做成一整行”、时间要“自动进位”、查看时“选中颜色去掉”，以及打开窗口“不使用共享画布外壳，不支持拖拉”。仅复用行 UI 会丢失展开箭头、展开状态、详情槽位和节点壳交互；但在详情槽位里再次渲染节点 header 会形成“节点行 + 重复节点头”的双层结构，节点名和指标上下排列也会破坏追踪列表的横向扫描效率，毫秒数过大则降低扫读效率，额外选中态颜色会制造不存在的选择语义，脱离共享 side dock 会丢失画布窗口统一拖拉能力，导致右侧节点详情和左侧对话日志追踪体验不一致。

适用场景：后续调整 `debug-console`、预览日志、工作流追踪、节点运行详情、LLM route / fusion trace 子树时，共享组件应覆盖 row + trigger + expanded detail slot；节点详情内容可作为 children 传入。
