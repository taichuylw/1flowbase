# Deliver Only After Real Verification

## Rule

涉及前端可见效果、接口载荷或用户已经反馈“重启后仍无变化”的修复，交付前必须先拿到可复用证据：目标入口测试、相关组件测试、接口回归或运行态检查至少覆盖用户实际看到的路径。

## Reason

用户关心的是“重启后打开实际页面是否变化”，不是代码是否看起来改对了。只修局部组件或只跑间接测试，容易漏掉真实入口没有接入、缓存数据仍走旧路径、接口仍一次返回大 JSON 等问题。

## Applies When

- 用户反馈实现后重启仍无变化。
- UI 修复涉及多个入口，例如编辑器调试面板和应用日志页。
- 接口改动目标是减少首包体积或延迟加载大字段。
- 最终回复准备声明“完成”“已修复”“可以用了”。

## Practice

最终回复前先说明已覆盖的实际入口和命令结果；若全量构建或全量测试因无关既有问题失败，要明确标注阻塞点，不能把它算作本次通过证据。
