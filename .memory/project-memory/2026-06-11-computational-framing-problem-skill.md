# Computational Framing In Problem Framing

用户在 2026-06-11 确认：需求对齐阶段应加入通用的计算表达假设，而不是只作为前端数学 / 物理动效清单。

已确认方向：

- `problem-framing` 在输出方案前，先判断需求是否能被数学关系、算法、数据结构、状态机、图、队列、约束、概率、评分、调度或物理公式更清晰地表达。
- 前端命中时，重点是空间关系、操作反馈、程序化视觉和低成本动效。
- 后端命中时，重点是状态流转、一致性、调度、查找、去重、排序、权限集合和约束校验。
- 输出时不列技术清单；写成方案假设、收益、代价、失败模式和验收证据。
- 若只是炫技、过度抽象、增加理解成本、性能风险高或无法验证，则明确不采用。
