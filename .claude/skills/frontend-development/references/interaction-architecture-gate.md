# Frontend Interaction Architecture Gate

## When To Enter

命中下面任一信号，就不要把它只当成普通页面细化：

- 调整导航、入口、页面层级或模块顺序
- 需要决定 `inline / Drawer / Inspector / Page / Dialog` 的详情或操作落点
- 同类对象开始出现不同点击结果、不同下钻路径或不同反馈位置
- 想把创建、编辑、批量管理、AI 执行塞进概览页或摘要卡片
- 用户说“这个页面不顺手 / 不直观 / 不知道去哪 / 有的能点有的不能点”
- 你已经说不清某个动作应该放在 `L0 / L1 / L2 / L3` 哪一层
- 同一页面在 1 到 2 天内反复移动入口、标题、卡片、状态栏、Tab、Modal 或操作按钮

## Goal

在进入普通需求整理和实现前，先判断这次是不是交互架构问题；如果是，先冻结最小交互规则，再决定是否需要完整 `frontend-logic-design` 诊断。

## Minimum Pass

命中本 gate 后，至少先回答这 6 个问题：

1. 首屏主任务是什么？
2. L1 详情模型是什么？为什么是它？
3. L2 管理入口在哪里？如何到达？
4. L3 执行动作放在哪里？是否需要脱离当前上下文？
5. 同类对象的点击和下钻行为是否一致？
6. 操作反馈落在哪里？是 inline、当前页、侧栏还是独立执行页？
7. 最近是否已经围绕同一页面发生多次 churn？如果是，先补 skill / 规则缺口，再继续局部改 UI。

## Mini Diagnostic Template

在回复用户时，先补一段很短的交互架构快审，再继续需求整理或实现：

```md
## 交互架构快审

- 首屏主任务：用户进入页面后第一件要完成的事是什么
- L0 概览：当前页直接展示哪些摘要，哪些不能塞进来
- L1 聚焦：单条详情统一走什么模型（`Drawer / Inspector / 展开面板`）
- L2 管理：全量列表、搜索、筛选、批量操作从哪里进入
- L3 执行：复杂执行 / AI 协作从哪里触发，是否跳出当前上下文
- 一致性规则：同类对象是否统一点击结果、统一反馈位置
- 热点预防：这次是否暴露了应更新的 frontend skill、AGENTS 或样式门禁
- 明确建议：这次是普通页面细化，还是已经进入结构性问题
```

## Escalate To Full `frontend-logic-design`

出现下面任一情况，就不要停留在 mini gate，直接升级到完整 `frontend-logic-design`：

- 需要重组导航树、页面边界或模块分组
- 需要用 `L0 / L1 / L2 / L3` 全量审计多个区域或多个卡片
- 需要做 MECE 检查，确认分类是不是互斥且穷尽
- 同一页存在 2 组以上对象，且点击、详情、查看全部、执行动作规则互相打架
- 你无法用一句话解释“为什么这个动作在这里，而不是在别处”

## Continue After Gate

- 如果 mini gate 已经把主路径、详情规则和反馈位置说清楚，就回到 `requirement-refinement.md` 继续整理需求。
- 如果发现问题本质是结构性设计，而不是普通页面实现，切到 `frontend-logic-design`，先诊断再落实现。
