# Frontend Anti-Patterns

| 反模式 | 风险 | 替代方式 |
| --- | --- | --- |
| 把外部视觉样本直接当项目规范 | 当前产品语法被外部品牌风格覆盖 | 以 `DESIGN.md` 为准，外部样本只局部借鉴 |
| 页面 God Component | 一个页面改动牵动所有逻辑 | 按变化原因拆出局部状态、局部视图、协议适配 |
| 状态四处散落 | Bug 难追、行为不稳定 | 先定义状态归属，再决定放置层级 |
| 为单次使用强行抽象 | 抽象先于需求，后续更难改 | 保持内联，等第二个真实场景再抽 |
| 协议拼装和渲染混写 | 前端被接口细节污染 | 先在适配层做转换，再交给视图层 |
| 顺手造新交互语法 | 界面一致性被打碎 | 复用既有交互语法，必要时先找人确认 |
| 同类区域点击行为不一致 | 用户心智混乱 | 把问题上升到结构层处理，不做局部打补丁 |
| 把第三方组件内部样式链当自家 DOM 递归覆盖 | 改一处伤多处，原生组件布局和交互被意外打坏 | 优先 `theme token -> first-party wrapper -> explicit slot`，到第三方内部链就停 |
| 裸写 `.ant-*` 全局样式 | 所有 `Ant Design` 消费者 blast radius 失控 | 必须从自有 wrapper 出发，并限制在单一明确 slot |
| 为了调一个局部视觉问题修改第三方内部布局指标 | `display / line-height / padding / min-height` 被打乱后，很难发现回归 | 先回到容器层或主题层；确需改内部布局时单独说明并验证 |
| 只改导航文案不改路由真值层 | 看起来“像改好了”，实际入口、选中态、权限映射仍旧错误 | 导航文案、`route id`、`path`、选中态统一由单一配置源维护 |
| `home` 页越界承接应用内重内容 | 根路由失去“应用目录 + 创建入口”职责 | 回到固定 recipe，应用内编排、日志、API 和监控进入 `/applications/:id/<section>` |
| 在 Shell / Canvas 间混用 L1 模型 | 同类对象的详情路径不可预测 | Shell 列表统一 `Drawer`，Canvas 对象统一 `Inspector` |
| 状态色复用为类型色或装饰色 | 运行态语义失真，跨页面不可比对 | 类型 badge 保持中性，状态只表达系统真相 |
| 把桌面画布硬塞进小屏 | 产生半可用界面，破坏首屏优先级 | 移动端做摘要态和明确引导 |
