# Frontend Requirement Refinement

## When To Enter

命中下面任一信号，就不要直接实现：

- 用户只说“做个页面 / 优化一下 / 改成这种感觉”
- 用户提出页面开发、页面改版、模块级 UI 开发、页面结构调整
- 用户只给图片、截图、竞品页、Dribbble / Awwwards 链接
- 用户想借外部样本，但没有说明借结构、借视觉还是借交互
- 这是新页面、新流程、新视觉方向，但页面目标和成功标准还没落地

只有纯局部样式修补、像素级对齐、文案替换、颜色 / 间距微调且不改页面结构时，才可以不进入完整流程。

## Goal

把 UI / 页面请求收敛成一份可直接回复给用户的前端设计需求，然后默认直接进入实现。

图片和外部样本只能提供灵感、结构和节奏，不能直接替代当前项目的 `DESIGN.md`、页面 recipe 和状态语义。
如果这次改动先命中了入口、层级、详情容器或执行落点的交互架构决策，先读 `references/interaction-architecture-gate.md` 完成 mini 快审，再回到本文件。

想深入看提炼方法时，读 `references/extraction-framework.md`。
想直接套固定输出骨架时，读 `references/skill-template.md`。
想看实际写法时，读 `examples/` 下的示例。

## Hard Rules

- 新页面、新流程、外部视觉母本、图片驱动需求，先收敛需求，再默认实现。
- 页面 / UI 开发需求的“需求整理 / 需求细化”必须出现在回复里，不能只在内部完成。
- 需求整理必须包含页面交互设计：主路径、关键反馈、模块协作和操作落点不能缺席。
- 参考图必须拆成“可借”和“不可借”两部分，不能默认整份照搬。
- 视觉语言、状态颜色、组件边界最终以 `DESIGN.md` 和现有页面 recipe 为准。
- 如果页面 / 模块改动会改变入口、层级、L1 详情容器、L2 管理入口或 L3 执行动作落点，先跑 `interaction-architecture-gate.md`；命中结构问题再升级到 `frontend-logic-design`。
- 与用户沟通时，优先先给出设计需求草案并继续实现；只有真正阻塞的产品级分歧，才一次性集中提问。

## What To Extract

先从请求里提炼这几类信息：

1. 页面目标
   - 这个页面服务谁
   - 用户进来后最重要的动作是什么
   - 什么结果算完成

2. 页面骨架
   - 是列表页、详情页、工作台、设置页、配置页，还是混合页
   - 顶部、侧栏、主区、辅助区分别承载什么
   - 哪些模块是必须项，哪些只是参考图里的装饰项

3. 页面交互
   - 用户从进入页面到完成目标的主路径是什么
   - 模块之间如何联动，操作后反馈落在哪里
   - 哪些只是内容承载区，哪些模块真的驱动下一步动作
   - 首屏主任务、L1 详情模型、L2 管理入口、L3 执行动作是否已经说清楚
   - 同类对象的点击和反馈行为是否一致

4. 关键对象与状态
   - 页面主对象是什么
   - 需要哪些状态：`loading / empty / error / success / running / selected / draft`
   - 哪些状态是当前项目已有语义，哪些只是参考图表现

5. 参考图借鉴边界
   - 借布局骨架
   - 借信息层级
   - 借卡片/列表/工具条的组合方式
   - 借交互节奏，但不直接搬交互契约
   - 不直接借品牌色、文案、图标和第三方产品气质

## Default Output Format

在实现前，先向用户输出一个设计需求草案，随后默认继续实现。这段内容是直接回复给用户 / 客户的，不是内部注释。推荐固定结构：

1. 任务理解
   - 我理解你要做的页面是什么，用来解决什么问题。

2. 参考图拆解
   - 这张图里我建议借哪些东西。
   - 这些东西不能直接照搬，必须回到当前项目规范。

3. 设计需求草案
   - 页面目标
   - 主要用户
   - 页面结构
   - 关键模块
   - 页面交互
   - 交互架构决策（如本轮命中 gate）
   - 关键动作
   - 关键状态
   - 视觉约束

4. 明确建议
   - 给出默认推荐方向，并说明为什么。

如果仍有阻塞实现的问题，再追加：

5. 待确认问题
   - 一次性列出会影响方向的决策问题，让用户统一回答。

## Minimum Output For Clear UI Requests

如果任务已经比较明确，但仍属于页面 / 模块 UI 开发，不要跳过需求整理。至少给出下面四段简版回复：

1. 任务理解
   - 这次要改哪个页面 / 模块，目标是什么。

2. 改动范围
   - 这次涉及哪些结构、模块、交互、状态，不涉及什么。

3. 需求细化
   - 关键动作、页面交互、关键状态、视觉约束各是什么。
   - 如果改动涉及交互架构决策，再补一句说明 L1 / L2 / L3 和反馈落点。

4. 明确建议
   - 说明推荐实现方向，并明确“我会按这版直接开始实现”。

## Suggested Questions

只有存在阻塞实现的分歧时，才集中确认这些问题：

- 页面名称和主要对象是什么
- 用户最常做的 1 到 2 个动作是什么
- 参考图里你最想继承的是结构、视觉，还是交互节奏
- 哪些模块是必须有的，哪些只是“像那个图一样有感觉”
- 这个页面需要覆盖哪些状态和异常场景
- 视觉上必须继承 `DESIGN.md` 的哪些规则，哪些允许局部借鉴外部样本

## Example Framing

如果用户只丢一张图，可以先这样收敛：

> 我会把这张图当成结构和信息节奏参考，不直接复刻它的品牌皮肤。先按当前项目的 `DESIGN.md` 回收视觉语言，整理一版页面目标、主路径和模块划分，然后直接开始实现；如果过程中存在阻塞决策，我会一次性列给你。
