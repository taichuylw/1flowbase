# Frontend Communication Gate

## Run The Interaction Architecture Gate First

如果任务涉及入口、层级、详情容器、`查看全部`、AI 执行动作或同类对象行为统一，先读 `interaction-architecture-gate.md` 做 mini 诊断，再判断是否进入下面的沟通门槛。

## Ask First

下列改动先与人沟通，再实现：

| 变更类型 | 先沟通 | 原因 |
| --- | --- | --- |
| 新页面或页面 recipe 调整，且存在阻塞性产品分歧 | Yes | 改变任务域边界和页面语法 |
| 新流程，且核心路径无法自行判断 | Yes | 改变任务路径和预期 |
| 交互流调整，且会改变既有行为契约 | Yes | 改变用户行为契约 |
| 入口 / 层级 / L0-L3 / 详情容器调整，且需要新增或改写规则 | Yes | 改变交互架构，不只是页面细化 |
| L1 模型调整 | Yes | 会影响 Drawer / Inspector 统一规则 |
| 状态语义调整 | Yes | 会影响全站运行态一致性 |
| 视觉方案调整或引入外部视觉母本，且借鉴边界无法自行收敛 | Yes | 容易打碎统一视觉语言 |
| 页面内 AI 协作层能力 | Yes | 属于产品级行为，不只是界面实现 |

## Usually Safe To Do Directly

下列改动通常可直接按 Skill 执行：

| 变更类型 | 通常可直接做 | 前提 |
| --- | --- | --- |
| 局部 bugfix | Yes | 不改变既有页面 recipe 和交互契约 |
| 同类页面内复用已有组件 | Yes | 不引入新交互规则 |
| 职责收敛 | Yes | 只是把混杂实现拆清楚 |
| 明显重复代码整理 | Yes | 没有过早抽象 |
| 在既有视觉基线内补齐缺失状态或响应式 | Yes | 不改动语义映射和页面边界 |

## Escalate Immediately

出现以下信号时，不要继续闷头写：

- 同类对象要采用不同点击行为
- 需要新增第三种 L1 详情模型
- 你说不清首屏主任务、L1 详情、L2 管理入口和 L3 执行动作分别放在哪里
- 想把 `awesome-design-md` 或别的视觉样本直接当成当前项目规范
- `home` 开始承接具体应用的完整编排、完整日志、API 正文、监控或说明性文案
- 页面逻辑里开始出现大量协议转换、拼接和条件分发
- 你已经说不清“这段状态到底归页面、组件还是共享层”

## Reply With Requirement Refinement Before Implementing

当任务属于页面 / UI 开发需求，且不是纯局部样式修补时，先在回复里做需求整理，再默认继续实现：

1. 明确页面目标、目标用户、核心动作和成功标准。
2. 说清本次改动范围：页面结构、关键模块、页面交互、关键状态、视觉约束。
3. 如果有参考图，说明哪些只借结构 / 信息层级 / 节奏，哪些视觉语言必须回到 `DESIGN.md`。
4. 先把这份设计需求草案直接回复给用户；只有仍有阻塞实现的分歧时，再一次性集中提问。

### Minimum Bar

- 新页面、页面改版、信息层级调整、关键模块改造、图片 / 外部样本驱动：使用完整需求草案。
- 明确范围的页面 / 模块 UI 开发：至少给简版需求整理，不能直接跳过。
- 只有纯局部样式修补、像素级对齐、文案替换、颜色 / 间距微调且不改变页面结构时，才可以不走完整需求整理。
