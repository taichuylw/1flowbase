# Frontend Requirement Extraction Framework

## Goal

把“模糊目标 / 图片参考 / 外部样本”翻译成当前项目可执行的页面需求，而不是把参考图直接当需求抄过来。

## The Extraction Order

先按下面顺序提炼：

1. 问题
   - 这个页面在解决什么问题
   - 不做这个页面，当前用户卡在哪里

2. 对象
   - 页面主对象是什么
   - 用户围绕哪个对象查看、筛选、配置、执行

3. 动作
   - 用户最关键的 1 到 2 个动作是什么
   - 哪些按钮、入口、操作必须优先露出

4. 交互
   - 用户从进入页面到完成动作的主路径是什么
   - 关键反馈落在哪里：`Drawer / Inspector / inline / toast / modal`
   - 模块之间如何联动，避免只剩内容堆积

5. 状态
   - 页面必须覆盖哪些状态：`loading / empty / error / success / running / selected / draft`
   - 哪些状态来自项目现有语义，哪些只是参考图表现

6. 骨架
   - 这是列表页、设置页、详情页、工作台还是混合页
   - 顶部、侧栏、主区、辅助区分别承载什么

7. 借鉴边界
   - 参考图只借结构、层级、节奏、组件组合
   - 视觉语言、品牌色、状态色、交互契约回到 `DESIGN.md`

## Image-Led Translation

看到图片时，优先拆这四层：

1. 信息层级
   - 页面先让人看到什么，再看到什么

2. 布局语法
   - 左右栏、顶部工具条、卡片分组、分段列表、辅助区位置

3. 组件关系
   - 搜索、筛选、提示条、卡片、操作区如何组合

4. 交互节奏
   - 页面入口、主操作、反馈位置和后续动作如何衔接

5. 不可照搬项
   - 品牌色
   - 文案
   - 图标
   - 第三方产品语气
   - 与当前项目冲突的状态表达

## Output Rule

先给用户一版设计需求草案，然后默认继续实现。

只有当下面问题仍然无法自行判断时，才集中提问：

- 页面主对象无法判断
- 核心动作有两个以上互相冲突的方向
- 参考图借鉴边界会改掉当前产品级语义
- 新流程会引入新的 L1 模型或新的状态语义
