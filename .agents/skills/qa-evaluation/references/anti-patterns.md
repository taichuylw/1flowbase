# QA Evaluation Anti-Patterns

| 反模式 | 风险 | 正确做法 |
| --- | --- | --- |
| 把 QA 做成另一个开发 Skill | 评估和修复混在一起，失去独立视角 | 默认只出报告和修正方向 |
| 没有证据就下结论 | 结论不可复查，容易误导决策 | 先跑验证、看截图、走流程，再下判断 |
| 把 QA 报告写成代码审查 | 只盯代码细节，忽略项目语义和用户路径 | 按范围、结论、Findings、风险输出 |
| 局部任务直接升级成全量审计 | 成本失控，打断开发节奏 | 默认 `task mode`，全量模式必须用户明确授权 |
| 只挑视觉毛病，不看契约和状态 | 抓不到真正高风险问题 | 同时检查 UI、流程、API、状态和边界 |
| 前端为了好用把后端字段改成另一套业务字段名 | 人和 AI 都会失去同一契约真相源 | 沿用 DTO / 领域字段名；旧字段兼容必须标记 `@field-contract-compat` 并进 warning |
| 只看当前改动点，不看变化传播 | 共享改动的回归风险被漏掉 | 抽查其他消费者和上下游依赖 |
| 把“代码看起来合理”当通过 | 纸面通过，实际行为未验证 | 以真实证据替代主观判断 |
