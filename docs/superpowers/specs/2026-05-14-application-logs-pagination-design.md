# 应用运行日志分页与默认排序设计

日期：2026-05-14

状态：待用户审阅

## 背景

当前应用运行日志页存在三个直接问题：

1. 日志列表接口一次返回当前应用的全部运行记录，前端没有分页。
2. 表格底部没有分页器，剩余高度虽然被列表容器占住，但在数据量较少时会形成明显空白。
3. 页面虽然支持本地排序切换，但分页、排序和查询真值没有统一落在后端接口上。

现状代码事实：

- 后端 `GET /api/console/applications/{id}/logs/runs` 当前直接返回 `Vec<FlowRunSummaryResponse>`。
- PostgreSQL 读取方法当前按 `created_at desc, id desc` 查询，也就是“最新在前”。
- 前端 `ApplicationRunsTable` 使用 `antd` `Table`，并且 `pagination={false}`。

## 目标

本轮把应用运行日志页改成标准分页列表，固定以下行为：

1. 默认每次查询 `20` 条。
2. 默认按创建时间倒序显示，最新记录排在最前。
3. 前端显示分页器，切页重新请求服务端数据。
4. 列表区域继续占用剩余高度，但底部由表格分页器承担空间，不再表现为大块无意义空白。

## 非目标

本轮不做以下事情：

1. 不新增无限滚动。
2. 不增加多字段服务端排序能力，只固化默认创建时间倒序。
3. 不新增关键字搜索和时间筛选以外的新筛选维度；现有关键字搜索与时间筛选如果为保证分页语义需要下推后端，允许作为本轮配套改动一起落地。
4. 不改运行详情浮窗和对话日志浮窗交互。

## 方案

### 1. 后端接口改为分页响应

把日志列表接口从“返回数组”改成“返回分页对象”。

建议 query 参数：

- `page`：从 `1` 开始，默认 `1`
- `page_size`：默认 `20`，本轮前端固定传 `20`

建议返回结构：

```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 137
}
```

排序真值固定为：

```text
order by created_at desc, id desc
```

这样前端无论初次进入还是切页，最新运行都会稳定出现在前面，并且不会因为本地切片而出现“第一页并不代表最新 20 条”的假象。

### 2. 存储层增加 total + limit/offset 查询

当前 repository 只有 `list_application_runs(application_id)`，需要升级为带分页输入的查询。

预期能力：

1. 查询总数 `total`。
2. 使用 `limit` 和 `offset` 读取当前页数据。
3. 继续返回现有 `ApplicationRunSummary` 字段集合，不改列表行字段定义。

优先保持单一职责：

- `control-plane` / repository trait 增加分页输入输出结构；
- `storage-durable/postgres` 实现 SQL `count(*)` 与分页读取；
- `api-server` route 负责 query 参数解析和响应序列化。

### 3. 前端查询状态改为页码驱动

前端日志页新增本地分页状态：

- `page`
- `pageSize = 20`

`react-query` 的 query key 要带上 `page` 和 `pageSize`，避免不同页复用同一缓存。

切页行为：

1. 用户点击分页器。
2. 更新当前页码。
3. 重新请求后端接口。
4. 表格显示当前页记录。

默认排序行为：

1. 初始查询始终请求第一页。
2. 默认按创建时间倒序。
3. 现有排序下拉如果继续保留，本轮至少要保证“创建时间”仍是默认项，且不会打破“最新在前”的默认加载行为。

如果当前“更新时间”本地排序与服务端分页组合后会制造歧义，本轮应优先收窄行为：

- 列表默认只保证创建时间倒序；
- 排序器可以暂时仅保留“创建时间”或调整文案，避免出现“当前页内本地排序”冒充全量排序的错误语义。

### 4. 表格使用 Ant Design 自带分页区

`ApplicationRunsTable` 继续使用 `antd` `Table`，但不再 `pagination={false}`。

改为受控分页配置：

- `current`
- `pageSize`
- `total`
- `showSizeChanger: false`

本轮前端固定每页 `20` 条，不开放用户改 page size。

分页器放在表格底部，承担底部空间。这样日志区仍然吃满剩余高度，但用户看到的是“表格内容 + 分页器”的完整列表结构，而不是一整块空白底板。

## 交互与数据语义

### 分页与筛选关系

当前关键字搜索和时间筛选是前端本地行为。分页改成服务端后，这两者会天然变成“只在当前页数据内过滤”，这不是真正的全量筛选。

因此本轮实现时有两个可接受落点，推荐第一个：

1. 推荐：把关键字搜索和时间筛选也同步下推到后端接口，保持“筛选作用于全量结果”的正确语义。
2. 退一步：如果本轮不下推后端，则在交付中明确这两项暂时只作用于当前页，并在 UI/文案上避免误导。

推荐理由：

- 一旦用了服务端分页，列表真值就不应继续由前端本地全量假设驱动；
- 否则页码、总数和筛选结果会在语义上互相打架。

本轮优先级顺序：

1. 服务端分页。
2. 服务端默认创建时间倒序。
3. 前端分页器。
4. 如果实现成本可控，把时间筛选和关键字搜索一并下推。

### 空白利用

本轮“利用剩余空间”的定义不是把少量行强行拉伸撑满，而是：

1. 日志列表容器继续占满剩余可用高度。
2. 表格主体在容器内正常显示。
3. 底部分页器固定可见。
4. 有较多数据时表格区滚动；数据较少时底部由分页区与表格结构共同占位，而不是只剩纯空白。

## 测试

实现前先写失败测试。

前端最小测试：

1. 日志页首次加载会按 `page=1`、`page_size=20` 请求。
2. 表格使用分页器，并显示总数/页码对应行为。
3. 点击下一页会触发新的 query 参数并刷新列表。
4. 默认排序对应“最新在前”的返回顺序，不被本地排序反转。

后端最小测试：

1. route 支持 `page`、`page_size` 参数。
2. route 返回分页对象而不是裸数组。
3. 默认 `page=1`、`page_size=20`。
4. 多条运行记录时，返回顺序为 `created_at desc, id desc`。
5. `total` 正确反映当前应用全部运行数。

## 验收证据

交付前最小证据链：

1. 前端日志页定向测试通过。
2. 后端日志列表 route / repository 定向测试通过。
3. 如改动命中样式边界，`page.application-logs` style-boundary 通过。
4. 浏览器级确认日志页：
   - 首屏默认看到最新记录；
   - 每页 20 条；
   - 分页器可切页；
   - 日志区底部不再只剩一整块裸空白。

## 停止条件

出现以下任一情况时先停下确认：

1. 当前前端关键字搜索和时间筛选若不一起下推后端，会导致分页语义明显错误。
2. 现有 API consumer 不止日志页一处，接口从数组改为分页对象会引发额外连锁改动。
3. 当前 route 或 repository 测试夹具默认假设“返回全量数组”，导致 blast radius 超出本轮。
